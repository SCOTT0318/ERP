# 4단계. 데이터베이스 모델 설계

## 목적

자재관리 ERP의 핵심 데이터 구조를 Django Model 기준으로 설계한다.

ERP는 데이터 정합성이 중요하다.

특히 재고관리에서는 현재고와 입출고 이력이 반드시 일치해야 한다.

## 기본 설계 원칙

- 현재고와 입출고 이력을 분리한다.
- 입고, 출고, 조정, 이동은 모두 이력으로 남긴다.
- 데이터는 실제 삭제하지 않고 비활성화 처리한다.
- 모든 주요 변경 작업은 변경이력에 기록한다.
- 향후 발주, 생산, 결재 기능과 연결 가능하도록 확장성을 확보한다.

## 핵심 모델 설명

- material_code
- material_name
- material_type
- category
- spec
- unit
- safety_stock_qty
- default_warehouse
- is_active
- is_deleted
- created_at
- created_by
- updated_at
- updated_by

- RM-000001
- SUB-000001
- WIP-000001
- FG-000001
- CONS-000001

- 출고취소

## 삭제 처리 방식

ERP에서는 데이터를 물리 삭제하지 않는다.

다음 필드를 사용한다.

- is_deleted
- deleted_at
- deleted_by

삭제 요청 시 실제 delete를 실행하지 않고 is_deleted 값을 true로 변경한다.

## Django Migration 원칙

- 모델 변경 후 makemigrations 실행
- migrate 실행 전 변경 내용 검토
- 운영 DB에는 검증 후 반영
- 마이그레이션 파일은 Git으로 관리
- 수동 DB 수정 최소화

## 인덱스 설계

입출고 이력과 현재고 조회가 많으므로 주요 필드에 인덱스를 적용한다.

- Material.material_code
- Material.material_name
- StockBalance.material
- StockBalance.warehouse
- StockBalance.location
- StockTransaction.transaction_date
- StockTransaction.material
- StockTransaction.transaction_type

## 향후 확장 고려 필드

나중에 발주, 생산, 결재와 연결하기 위해 다음 참조 구조를 고려한다.

- ref_doc_type
- ref_doc_id

예시

- ref_doc_type = PURCHASE_ORDER
- ref_doc_id = 1001
- ref_doc_type = WORK_ORDER
- ref_doc_id = 2001
